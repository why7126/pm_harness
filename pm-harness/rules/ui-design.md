# UI 设计系统与前端体验规范

## 0. 规则定位 `[通用]`

本文件定义 `{PRODUCT_NAME}` 的 UI 设计定位、视觉系统、组件复用、页面布局、交互状态、响应式适配、可访问性和 AI 修改边界。凡涉及 Web、移动端、微信小程序、桌面端、管理后台、控制台、可视化页面或设计系统的变更，均必须遵守本规范。

AI 在执行以下任务前必须先阅读本文件：

- 新增或修改页面、组件、布局、交互、视觉样式、图标、插画、图片和主题。
- 修改 Design Token、颜色、字号、间距、圆角、阴影、动效和响应式断点。
- 引入或替换 UI 组件库、图标库、样式方案、设计系统构建方式。
- 实现登录页、首页、控制台、列表、详情、表单、弹窗、上传、图表、媒体预览等核心体验。
- 进行前端重构、适配、视觉还原、截图验收或多端一致性调整。

## 2. 设计定位 `[通用 + 个性化]`

产品定位为：`{PRODUCT_POSITIONING}`。

目标用户为：`{TARGET_USERS}`。

支持端为：`{SUPPORTED_CLIENTS}`。

视觉风格为：`{DESIGN_STYLE}`。

设计原则为：`{DESIGN_PRINCIPLES}`。

初始化生成时必须明确：

- 这是工作台、后台管理、内容消费、创作工具、数据分析、交易转化、移动端应用还是多端产品。
- 用户主要任务是浏览、检索、录入、审批、分析、配置、创作、协作还是运维。
- 界面应偏信息密度、可读性、沉浸感、品牌表现、转化效率、可视化洞察还是高频操作效率。
- 首屏必须呈现的产品信号、核心能力和下一步动作。
- 哪些页面需要高度还原设计稿，哪些页面遵循通用组件规范即可。

UI 技术栈为：`{UI_STACK}`。

| 层级 | 项目配置 |
| --- | --- |
| 前端框架 | `{FRONTEND_FRAMEWORK}` |
| 样式方案 | `{STYLE_SYSTEM}` |
| 组件库 | `{COMPONENT_LIBRARY}` |
| 图标库 | `{ICON_LIBRARY}` |
| 状态管理 | `{STATE_MANAGEMENT}` |
| 表单方案 | `{FORM_STACK}` |
| 图表/可视化 | `{CHART_STACK}` |
| 动效方案 | `{MOTION_STACK}` |
| 测试/验收 | `{UI_VERIFY_COMMANDS}` |

通用要求：

- 优先使用项目已启用的组件库、图标库、Token 和布局系统。
- 不得为单个页面随意引入新的 UI 库、CSS-in-JS 方案、图标库或动效库。
- 若需要引入新依赖，必须说明复用价值、体积影响、主题兼容性和维护成本。
- 前端 UI 规则必须与 `rules/coding.md`、`rules/language.md`、`rules/testing.md` 保持一致。

## 4. Design Token `[条件启用]`

当项目启用设计系统、主题、白标、多端统一视觉或 Tailwind/组件库主题扩展时启用本节。

Design Token 单一事实源为：`{DESIGN_TOKEN_SOURCE}`。

Token 同步命令为：`{TOKEN_SYNC_COMMAND}`。

推荐 Token 分类：

| Token 类型 | 示例内容 |
| --- | --- |
| 色彩 | 背景、文本、边框、品牌、语义状态、图表色 |
| 字体 | 字体栈、字号、字重、行高、字距 |
| 间距 | 页面边距、组件内边距、网格间距、区块间距 |
| 圆角 | 按钮、输入框、卡片、弹窗、标签、媒体容器 |
| 阴影 | 浮层、弹窗、导航、悬浮卡片 |
| 动效 | 时长、缓动、进入/退出、hover/focus |
| 断点 | 移动端、平板、桌面、大屏 |
| z-index | 顶栏、侧栏、下拉、弹窗、Toast、引导层 |

规则：

- Token 应作为视觉系统的单一事实源，不得在业务组件中散落硬编码颜色、字号、圆角和阴影。
- 新增颜色必须先判断是否可映射到语义 Token。
- 业务组件应优先使用语义 Token，例如背景、文本、边框、主按钮、危险状态、成功状态。
- 修改 Token 后必须同步生成产物、预览页、组件示例和视觉回归基准。
- 未建立 Token 系统时，应在本节标记 `待完善`，并至少约定颜色、字体、间距和圆角的最小规范。

## 5. 色彩系统 `[通用 + 个性化]`

色彩系统必须基于 `{DESIGN_STYLE}` 和 `{DESIGN_TOKEN_SOURCE}` 生成。

推荐定义：

| 类型 | 必须说明 |
| --- | --- |
| 页面背景 | 默认页面、次级页面、浮层、侧栏、表格背景 |
| 文本颜色 | 主文本、次文本、弱文本、禁用文本、反色文本 |
| 品牌色 | 主色、强调色、hover、active、focus、浅底色 |
| 语义色 | 成功、警告、错误、信息、处理中、禁用 |
| 边框/分割线 | 默认、强调、hover、focus、危险 |
| 图表色 | 分类色、连续色、警戒线、空状态 |

要求：

- 品牌色应克制使用，优先用于主 CTA、关键状态、激活态和核心识别。
- 状态色必须语义一致，不得同一颜色在不同页面表达冲突状态。
- 文本与背景对比度必须满足 `{ACCESSIBILITY_TARGET}`。
- 禁止在同一页面引入大量无 Token 管理的临时颜色。

## 6. 字体、间距与布局密度 `[通用 + 个性化]`

字体系统必须定义：

- 字体栈和可选品牌字体。
- 标题、页面标题、区块标题、卡片标题、正文、辅助文字、标签、数字的字号和字重。
- 行高、段落间距、表格密度、数字对齐规则。
- 中英文、数字、单位、代码、金额、日期时间的展示规则。

间距系统必须定义：

- 页面左右边距、内容最大宽度、区块间距。
- 表单项间距、表格行高、卡片内边距、工具栏间距。
- 圆角、边框、阴影、分割线和视觉层级。

规则：

- 后台、CRM、运维、管理类界面应优先信息密度和可扫描性。
- 内容、品牌、作品、商品展示类界面可以更强调首屏视觉和媒体表现。
- 按钮、输入框、标签、卡片、表格的尺寸不得因动态内容导致布局跳动。
- 文本不得溢出、遮挡、重叠；长文本必须有换行、截断、Tooltip 或展开策略。

## 7. 组件使用优先级 `[通用 + 个性化]`

组件库为：`{COMPONENT_LIBRARY}`。

组件使用优先级：

1. 项目 Design System 基础组件。
2. 项目业务通用组件。
3. 当前模块内已有组件。
4. 经过评估后新增组件。

新增组件规则：

- 新组件必须放入项目约定目录，并说明是否可复用。
- 组件 API 应稳定、命名清晰、状态完整，避免把页面业务强塞进基础组件。
- 基础组件不得引用业务接口、路由、权限或模块私有状态。
- 业务组件应复用基础组件、Token、图标和通用交互。
- 不得复制粘贴一个近似组件来规避抽象；但也不得为了单个页面过度抽象。

## 8. 核心组件规范 `[通用 + 个性化]`

生成项目文档时，应根据 `{CORE_COMPONENTS}` 定义核心组件规范。

推荐覆盖：

| 组件 | 必须定义 |
| --- | --- |
| Button | 尺寸、层级、loading、disabled、icon、危险操作 |
| Input/Form | label、placeholder、错误、帮助文本、必填、校验 |
| Select/Dropdown | 空状态、搜索、禁用、长文本、键盘操作 |
| Table/List | 密度、排序、筛选、分页、批量操作、空状态 |
| Card | 信息层级、媒体比例、操作区、hover、选中态 |
| Dialog/Drawer | 标题、说明、关闭、确认、危险操作、焦点管理 |
| Toast/Message | 成功、失败、警告、重试、持续时间 |
| Tabs/Segmented | 激活态、内容切换、响应式 |
| Badge/Tag | 状态语义、颜色、尺寸、可关闭 |
| Upload | 文件态、进度、失败、重试、预览、权限 |
| Navigation | 顶栏、侧栏、面包屑、移动端折叠 |

组件状态必须覆盖：

- default、hover、active、focus、disabled、loading、selected、error、empty、readonly。
- 需要键盘操作的组件必须有可见 focus 状态。
- 危险操作必须有明确视觉层级和二次确认策略。

## 9. 页面结构与路由布局 `[通用 + 个性化]`

路由与布局矩阵为：`{ROUTE_LAYOUTS}`。

核心页面为：`{CORE_PAGES}`。

页面结构必须定义：

- 哪些页面使用全局导航、侧边栏、顶部栏、底部栏、面包屑、工具栏。
- 登录、公开页、工作台、详情页、编辑页、全屏页、嵌入页的布局差异。
- 列表页、详情页、表单页、配置页、数据看板、媒体页、错误页的通用结构。
- 首屏内容、主要操作、次要操作、返回路径和空状态。
- 页面级 loading、error、empty、no permission、not found 的呈现方式。

规则：

- 页面结构优先服务核心任务，不得把营销式大 Hero 套到高频工作台和后台管理界面。
- 管理类页面应优先工具栏、筛选、表格、批量操作、详情抽屉和状态反馈。
- 内容展示类页面应优先媒体质量、阅读节奏、筛选导航和清晰 CTA。
- 多步骤流程必须展示进度、可返回路径、保存状态和失败恢复方式。

## 10. 表单、表格与数据展示 `[条件启用]`

当项目包含管理后台、配置、录入、审批、数据分析或复杂业务操作时启用本节。

表单：

- 必填项、校验错误、帮助文本、异步校验、提交中、提交失败必须有明确状态。
- 危险或不可逆操作必须二次确认。
- 大表单应分组、分步或提供锚点导航。
- 表单默认值、单位、格式说明和服务端错误必须清晰可见。

表格：

- 表格必须定义列宽、对齐、排序、筛选、分页、空状态、批量操作和行操作。
- 数字、金额、百分比、日期时间应使用一致格式。
- 长文本应有截断、Tooltip、展开或详情入口。
- 批量操作必须展示选择数量和影响范围。

数据展示：

- 指标卡、图表、趋势、状态分布必须有标题、单位、时间范围和空数据策略。
- 图表颜色必须来自 Token 或图表色板。
- 数据异常、加载失败和权限不足不得显示误导性空图。

## 11. 媒体、图片与可视化 `[条件启用]`

当项目包含图片、视频、音频、文件预览、图表、地图、3D 或可视化场景时启用本节。

- 媒体容器必须定义比例、裁剪、占位、加载失败、懒加载和预览策略。
- 商品、作品、文件、视频等需要识别主体的页面，不得使用模糊、过暗或过度裁切的图片。
- 图表必须提供单位、图例、Tooltip、空状态、异常状态和可读颜色。
- 交互式可视化必须提供缩放、重置、加载、错误和无数据状态。
- 媒体上传、预览和下载必须与 `rules/media.md`、`rules/object-storage.md`、`rules/security.md` 保持一致。

## 12. 交互与反馈 `[通用]`

所有用户操作必须有明确反馈：

- API 调用需要 loading、禁用重复提交或幂等策略。
- 成功操作应反馈结果，失败操作应说明原因和可恢复动作。
- 长任务应展示进度、排队、处理中、失败、取消或重试。
- 页面跳转、保存、删除、导入导出、上传下载必须避免无响应。
- Toast 只用于轻量反馈；重要错误、阻塞流程和危险确认应使用页面内提示或弹窗。

交互状态：

- hover 用于提示可交互，不能作为唯一信息来源。
- focus 必须可见，键盘用户可完成核心流程。
- disabled 必须说明原因，或提供可达的说明入口。
- loading 不得遮挡不可恢复错误。

## 13. 响应式与多端适配 `[条件启用]`

响应式策略为：`{RESPONSIVE_POLICY}`。

当项目支持移动端、微信小程序、桌面端、大屏或多端 Web 时启用本节。

- 必须定义断点、布局变化、导航变化、表格降级、筛选收纳和操作入口。
- 移动端应优先单列、底部操作、可触达按钮和表单可用性。
- 桌面端应优先信息密度、快捷操作、侧栏导航和多列布局。
- 大屏应避免简单拉伸，需定义最大宽度、网格扩展或数据看板布局。
- 不支持的端必须在文档中明确，不得做半成品适配。

## 14. 可访问性与可用性 `[通用 + 个性化]`

可访问性目标为：`{ACCESSIBILITY_TARGET}`。

要求：

- 文本对比度、字号、焦点态、键盘导航、表单 label、错误提示必须可用。
- 图标按钮必须有可访问名称或 Tooltip。
- 图片、图表、状态图标应有替代文本、标题或说明。
- 颜色不得作为表达状态的唯一方式。
- 弹窗、抽屉、下拉、菜单必须处理焦点进入、焦点回收、Esc 关闭和遮罩点击策略。
- 动效不得影响可读性；必要时支持减少动效。

## 15. 文案、国际化与格式 `[条件启用]`

国际化策略为：`{I18N_POLICY}`。

- 界面文案应简洁、具体、面向用户动作，不写实现细节。
- 按钮使用动词，状态使用名词或短语，错误提示说明恢复路径。
- 日期、时间、金额、数字、单位、百分比、文件大小必须格式统一。
- 多语言项目不得在组件中硬编码展示文案，应使用项目约定的 i18n 方案。
- 专业术语、品牌名、模块名应与 `rules/language.md` 保持一致。

## 16. 主题、白标与客户定制 `[条件启用]`

主题和定制策略为：`{THEMING_POLICY}`。

当项目支持暗色模式、品牌主题、客户定制、私有化部署或白标时启用本节。

- 主题差异必须通过 Token、配置或主题包表达，不得在业务组件中硬编码客户分支。
- 客户定制必须明确哪些能力可配置、哪些能力需要独立组件、哪些能力不可变更。
- 私有化运行时配置不得破坏布局、权限、安全和可访问性。
- 定制页面必须保留统一的交互、错误、加载和验收标准。

## 17. 视觉验收与验证 `[通用 + 个性化]`

视觉验收来源为：`{VISUAL_ACCEPTANCE_SOURCE}`。

UI 验证命令为：`{UI_VERIFY_COMMANDS}`。

要求：

- 有设计稿、原型或 Golden Reference 的页面，必须以其为最高优先级视觉来源。
- UI 变更后必须运行类型检查、Lint、组件测试、截图验证或项目约定命令。
- 关键页面应通过浏览器实际访问确认，检查布局、交互、loading、empty、error、移动端和文本溢出。
- 修改 Design Token 或基础组件时，应检查多个代表页面，避免局部改动造成全局回归。
- 无法运行自动化验证时，必须说明人工验证范围和残余风险。

## 18. AI UI 修改规则 `[通用]`

AI 修改 UI 时必须：

- 先查找可复用组件、Token、样式工具、图标库和既有页面模式。
- 遵循项目组件优先级，不随意新增重复组件。
- 使用项目约定的 API Client、状态管理、表单方案和错误处理。
- 不硬编码颜色、密钥、客户名、真实数据和临时路径。
- 不把业务逻辑写进基础组件，不把基础组件做成单页专用。
- 不为后台工具做营销式页面，不为品牌展示页做过度密集的后台布局。
- UI 变更后运行 `{UI_VERIFY_COMMANDS}`；未知时标记 `待确认`。
- 涉及截图还原、3D、Canvas、图表、响应式和复杂交互时，必须进行实际渲染验证。

## 20. 初始化生成建议 `[通用]`

用于工程初始化生成 `ui-design.md` 时，建议按以下步骤处理：

1. 根据用户输入替换 `{PRODUCT_NAME}`、`{SUPPORTED_CLIENTS}`、`{UI_STACK}`、`{DESIGN_STYLE}`、`{COMPONENT_LIBRARY}`、`{ROUTE_LAYOUTS}`、`{CORE_PAGES}` 等占位符。
2. 保留所有 `[通用]` 章节，并将模板语气改为项目确定性约束。
3. 对 `[个性化]` 章节填入产品定位、目标用户、技术栈、Token 源、组件目录、验收命令；信息不足时标记 `待确认`。
4. 对 `[条件启用]` 章节按项目能力裁剪；无前端、无移动端、无媒体、无主题、无 i18n 的项目应删除或标记“不适用”。
5. 不得保留其他项目的品牌名、颜色、页面名、业务模块、客户名、截图路径或设计稿路径。
6. 与 `directory-structure.md`、`coding.md`、`api.md`、`language.md`、`media.md`、`security.md`、`testing.md` 保持一致。

## 21. 完成任务后检查清单 `[通用]`

- 是否优先复用了已有组件、Token、图标和页面模式。
- 是否没有硬编码未登记的颜色、字号、圆角、阴影和客户差异。
- 是否覆盖 loading、empty、error、disabled、focus、hover、selected 等状态。
- 是否文本不溢出、不重叠、不遮挡核心操作。
- 是否响应式和目标端适配符合 `{RESPONSIVE_POLICY}`。
- 是否前端安全、权限、媒体、对象存储和 API 调用与对应规则一致。
- 是否运行了 `{UI_VERIFY_COMMANDS}` 或说明无法运行的原因。
