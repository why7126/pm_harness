# UI 设计规范

## 1. 基本原则

- UI 任务必须先确认产品形态、目标用户和品牌方向。
- 管理端/运营工具强调效率、扫描、筛选、表格、表单和重复操作。
- 官网/展示端强调首屏识别、内容层级、视觉资产和转化路径。
- 移动端/小程序强调触控、弱网、单手操作和平台限制。
- 具体视觉风格在项目初始化后写入本文件，不得沿用模板占位描述。

## 2. Token 与资源地图

项目启用 Design System 时，必须明确：

```text
Design Token 单一事实源
CSS / Tailwind / 组件库映射
基础组件目录
业务组件目录
页面模板目录
预览或验收入口
```

修改 Token 流程：

```text
1. 若设计语义变更，先更新本文件
2. 更新 token 源文件
3. 同步 CSS / Tailwind / 组件库映射
4. 运行生成或同步命令
5. 更新预览页或视觉验收材料
```

## 3. 强制样式规则

- 新增/修改 UI SHOULD 使用 semantic token，避免裸 Hex 和散落的硬编码颜色。
- className 合并使用项目既有工具函数。
- 新组件优先复用组件库、复合 UI、业务组件和页面模板。
- 不得在页面层重复实现已有 Design System 组件。
- 有 prototype 或 Golden Reference 时，以 prototype / Golden Reference / acceptance / 本文件的顺序验收。

## 4. 页面结构摘要

| 场景 | 优先结构 |
|---|---|
| 管理端列表 | 筛选区 + 表格/卡片表格 + 分页 + 批量操作 |
| 管理端表单 | 分组表单 + 校验反馈 + 清晰保存/取消动作 |
| 展示端首页 | 品牌/产品首屏 + 搜索/筛选 + 核心内容 + CTA |
| 详情页 | 核心信息 + 媒体预览 + 属性/规格 + 操作区 |
| 移动端 | 单列信息流 + 固定关键操作 + 平台原生能力适配 |

## 5. 验收

UI 变更完成前必须说明：

```text
□ 是否影响 Web / 管理端 / 移动端 / 小程序
□ 是否修改 token 或组件库
□ 是否存在裸 Hex / 硬编码样式
□ 是否运行构建、截图、视觉检查或组件预览
□ 是否需要补充交互、可访问性或响应式测试
```
