# 测试治理与质量门禁规范

## 0. 规则定位 `[通用]`

本文件定义 `{PRODUCT_NAME}` 的测试分层、覆盖率目标、运行命令、数据隔离、Mock 策略和质量门禁。所有新增功能、行为变更、Bug 修复、接口变更、数据结构变更和安全策略变更，都必须同步评估测试影响。

AI 在执行以下任务前必须先阅读本文件：

- 新增或修改后端、前端、移动端、算法、脚本、SDK、任务队列或部署逻辑。
- 修改 API 契约、错误码、权限、数据模型、迁移脚本、对象存储、媒体处理或外部服务调用。
- 修复 Bug、调整业务规则、变更兼容性策略、修改测试框架或 CI 流程。
- 生成、删除、跳过、放宽测试，或修改覆盖率、质量门禁、Mock 规则。

## 2. 测试总原则 `[通用]`

- 行为变更必须有测试：新增功能、Bug 修复、接口变更、权限变更、数据变更必须补充或更新测试。
- 测试先于或同步于实现：推荐 TDD；至少在实现完成前补齐对应测试。
- 测试描述业务行为：测试名称和断言应表达用户可观察行为或稳定的模块契约。
- 测试分层清晰：单元测试验证局部逻辑，集成测试验证模块协作，契约测试验证接口协议，E2E 验证关键用户路径。
- 测试可重复：测试不得依赖执行顺序、真实时间、真实外部服务、生产数据、开发者本机状态。
- 测试快速反馈：默认测试命令应可在本地快速运行；慢速测试、E2E、性能测试应有独立命令和标记。
- 测试失败优先修复：不得通过删除断言、跳过测试、放宽安全规则或降低覆盖率来掩盖问题。

## 3. 覆盖率要求 `[通用 + 个性化]`

全局覆盖率目标为：`{COVERAGE_TARGET}`。

分层覆盖率目标为：`{COVERAGE_BY_LAYER}`。

推荐默认目标：

| 层级 | 默认要求 |
| --- | --- |
| 核心业务 Service/UseCase | 重点覆盖成功路径、业务错误、权限错误、边界条件 |
| Repository/DAO/数据访问 | 覆盖查询条件、分页、事务、异常、迁移兼容 |
| API/Controller/Router | 覆盖请求校验、鉴权、响应结构、错误码 |
| 前端组件/页面 | 覆盖关键交互、状态切换、表单校验、错误展示 |
| SDK/共享类型 | 覆盖序列化、类型转换、兼容性和边界值 |
| 脚本/工具 | 覆盖参数校验、失败路径、幂等性和输出格式 |

覆盖率规则：

- 新增代码应达到项目设定的覆盖率目标。
- 核心业务、权限、安全、计费、数据迁移、导入导出、删除操作应优先高覆盖。
- 允许对纯类型定义、常量、框架启动胶水代码设置合理排除，但必须在测试配置中显式说明。
- 降低覆盖率阈值、扩大排除范围、跳过失败测试必须经过人工确认。

## 4. 测试分层 `[通用 + 个性化]`

推荐测试分层：

| 测试类型 | 目标 | 默认目录 |
| --- | --- | --- |
| 单元测试 | 验证函数、类、组件、Service、工具函数的局部行为 | `tests/unit/` |
| 集成测试 | 验证 API、数据库、对象存储、队列、模块间协作 | `tests/integration/` |
| 契约测试 | 验证 OpenAPI、SDK、Webhook、外部接口契约 | `tests/contract/` |
| E2E 测试 | 验证关键用户路径、浏览器流程、端到端链路 | `tests/e2e/` |
| 兼容性测试 | 验证浏览器、设备、数据库、对象存储、部署形态 | `tests/compatibility/` |
| 性能测试 | 验证关键接口、任务、导入导出、媒体处理的性能边界 | `tests/performance/` |

生成项目文档时必须根据 `{TEST_STRATEGY}` 和实际目录调整测试分层，不存在的能力不得保留强制测试目录。

## 5. 测试用例结构 `[通用]`

测试用例推荐使用 Arrange-Act-Assert 模式：

```python
def test_create_resource_success(service, current_user):
    # Arrange
    request = build_create_request(name="example")

    # Act
    result = service.create_resource(request, current_user)

    # Assert
    assert result.id is not None
    assert result.name == "example"
```

通用要求：

- 测试名称应表达场景和期望，例如 `test_create_resource_returns_403_when_user_has_no_permission`。
- 一个测试只验证一个主要行为；复杂流程应拆成多个测试。
- 断言必须具体，不得只断言“不报错”。
- 对时间、随机数、ID、外部返回值应使用固定种子、Fake、Mock 或工厂生成。
- 测试注释应解释复杂场景原因，不应重复代码含义。

## 6. 必须覆盖的场景 `[通用]`

每个核心 Service、Repository、API、组件或任务至少评估以下场景：

1. 成功路径：有效输入、正常权限、正常依赖返回。
2. 输入校验错误：缺少必填、类型错误、格式错误、长度越界、枚举非法。
3. 业务逻辑错误：非法状态转换、重复提交、资源不存在、规则冲突。
4. 权限错误：未登录、权限不足、租户越权、资源不可见、角色不匹配。
5. 外部服务失败：超时、限流、网络错误、返回结构异常、第三方错误码。
6. 数据边界：空列表、空字符串、null、最大值、最小值、分页边界、并发冲突。
7. 安全边界：敏感字段脱敏、上传限制、签名校验、路径穿越、防注入。
8. 兼容边界：旧数据、旧接口、旧配置、可选字段、迁移前后行为。

## 7. TDD 与需求追踪 `[通用]`

- 修改行为前应先写或更新测试；如先实现，提交前必须补齐测试。
- 测试应能追踪到需求、Bug、OpenSpec change、用户故事或验收标准。
- 若项目使用 OpenSpec，测试应覆盖 specs 中的 Given/When/Then。
- Bug 修复必须先补充失败复现测试，再修复实现。
- 需求验收标准变化时，必须同步更新测试和测试文档。

推荐映射：

| 来源 | 测试要求 |
| --- | --- |
| `issues/requirements/*/acceptance.md` | 覆盖验收标准 |
| `openspec/changes/*/specs/*/spec.md` | 覆盖 Given/When/Then |
| `docs/standards/api-governance.md` / OpenAPI | 覆盖请求、响应、错误码 |
| Bug 记录 | 覆盖复现用例和回归用例 |

## 8. Mock、Stub 与外部依赖 `[通用 + 个性化]`

Mock 策略为：`{MOCK_STRATEGY}`。

通用要求：

- 单元测试必须隔离真实外部服务，例如支付、短信、邮件、LLM、语音、OCR、对象存储、搜索、工作流、Webhook、企业系统。
- Mock 返回值必须匹配真实接口结构，不得为了测试方便返回不可能出现的数据形态。
- Mock 断言应覆盖调用参数、调用次数、错误处理和超时路径。
- 集成测试可以使用本地容器、Fake 服务、测试账号或沙箱环境，但不得调用生产服务。
- 外部服务契约变化时，必须更新 Mock、契约测试和错误处理测试。
- 禁止在单元测试中访问真实网络、真实对象存储、真实数据库、真实密钥或生产数据。

## 9. Fixture、工厂与测试数据 `[通用 + 个性化]`

Fixture 策略为：`{FIXTURE_STRATEGY}`。

- Fixture 应表达业务语义，例如 `admin_user`、`readonly_user`、`sample_resource`、`expired_token`。
- 测试数据必须使用脱敏数据、合成数据或最小样例，不得使用生产数据。
- 测试数据应尽量由工厂函数生成，避免在多个测试中复制大段字典或 JSON。
- 固定样例文件应放在 `tests/fixtures/` 或项目约定目录，并说明用途。
- Fixture 不得隐藏关键断言；测试读者应能看出场景的输入条件。
- 测试账号、密钥、Token 必须是无效样例值或测试环境专用值。

## 10. 测试数据库与隔离 `[条件启用]`

当项目使用数据库、缓存、搜索引擎或状态存储时启用本节。

测试数据库策略为：`{TEST_DATABASE_STRATEGY}`。

- 测试不得连接生产数据库。
- 每个测试必须隔离数据，避免依赖测试顺序。
- 推荐使用事务回滚、临时数据库、容器数据库、内存数据库或独立测试 schema。
- 数据库迁移必须在测试环境验证，尤其是新增字段、默认值、索引、约束、数据回填和回滚策略。
- Repository/DAO 测试必须覆盖查询条件、分页、排序、权限过滤、空结果和异常路径。
- 如项目支持多数据库或国产数据库，应在 `tests/compatibility/` 中维护兼容性测试矩阵。

## 11. API 与契约测试 `[条件启用]`

当项目提供 HTTP API、RPC、Webhook、SDK 或 OpenAPI 文档时启用本节。

API 测试策略为：`{API_TEST_STRATEGY}`。

- 接口变更必须补充或更新集成测试。
- API 测试必须覆盖请求校验、认证、权限、成功响应、错误响应、错误码和响应结构。
- OpenAPI、SDK、前端 API Client、接口文档必须与测试保持一致。
- Webhook 必须测试签名验证、重放保护、非法来源、重复投递和失败重试。
- 向后兼容接口必须保留旧字段、旧错误码或旧路径的回归测试。

## 12. 前端测试 `[条件启用]`

当前端、管理台、H5、桌面端或可视化界面存在时启用本节。

前端测试栈为：`{FRONTEND_TEST_STACK}`。

- 类型检查、Lint、单元测试、组件测试应有明确命令。
- 关键组件必须覆盖渲染、交互、空状态、加载状态、错误状态、权限状态。
- 表单必须覆盖必填、格式、边界值、提交成功、提交失败、服务端错误展示。
- API 调用应 Mock 网络层，避免单元测试依赖真实后端。
- UI 行为变更应使用组件测试、截图验证、浏览器验证或 E2E 测试覆盖。
- 若当前项目暂未配置前端测试框架，必须至少保留类型检查、Lint 和手动验证记录，并在初始化文档中标记 `待完善`。

## 13. 移动端、微信小程序与桌面端测试 `[条件启用]`

当项目包含移动端、微信小程序、桌面端或多端客户端时启用本节。

移动端测试栈为：`{MOBILE_TEST_STACK}`。

- 必须覆盖核心页面、登录态、权限、表单、网络错误、弱网、缓存和版本兼容。
- 微信小程序应覆盖平台 API Mock、登录授权、页面跳转、分包、上传下载和基础兼容。
- 桌面端应覆盖本地文件、系统权限、自动更新、离线状态和窗口交互。
- 多端共享逻辑应优先沉淀到共享模块并进行单元测试。

## 14. E2E 与关键路径测试 `[条件启用]`

当项目存在完整用户流程、浏览器端到端验证、发布验收或回归套件时启用本节。

E2E 测试栈为：`{E2E_TEST_STACK}`。

- E2E 测试只覆盖关键路径，不替代单元测试和集成测试。
- 关键路径应覆盖登录、核心创建/编辑/删除、权限隔离、导入导出、上传下载、支付或审批等业务闭环。
- E2E 数据必须可重复初始化和清理。
- 浏览器测试必须避免固定等待，优先使用可观察状态、请求完成、元素可见等条件。
- 失败时应保留截图、视频、trace 或日志，方便定位。

## 15. 媒体、上传与对象存储测试 `[条件启用]`

当项目启用文件上传、媒体处理、对象存储、导入导出或下载预览时启用本节。

- 上传测试必须覆盖大小限制、扩展名、MIME Type、文件头、非法文件、空文件、重复上传。
- 文件名、路径、对象 Key 必须测试路径穿越、特殊字符、超长值和冲突处理。
- 下载、预览、签名 URL 必须测试权限、过期、资源不存在和跨租户访问。
- 媒体处理必须测试转码、封面、解析、超时、失败清理和临时文件删除。
- 对象存储测试应使用 Fake、本地容器、测试桶或 Mock 客户端，不得使用生产桶。

## 16. 安全测试 `[通用 + 个性化]`

安全测试策略为：`{SECURITY_TEST_STRATEGY}`。

- 必须测试未登录、权限不足、租户越权、资源不可见、管理接口越权。
- 必须测试输入校验、防注入、非法枚举、超长参数、危险路径、非法文件。
- 必须测试敏感字段脱敏、错误信息不暴露内部路径、日志不输出密钥。
- 必须测试公开端点清单，确保未声明公开的业务接口默认需要鉴权。
- 安全修复必须补充回归测试，防止后续放宽。

## 17. 性能与稳定性测试 `[条件启用]`

当项目有性能指标、批量导入导出、媒体处理、算法任务、长任务或高并发要求时启用本节。

性能测试策略为：`{PERFORMANCE_TEST_STRATEGY}`。

- 性能测试必须明确目标指标、数据规模、执行环境、预热方式和通过标准。
- 基准测试应与单元测试分开运行，避免影响默认反馈速度。
- 长任务、队列、批处理、媒体处理、外部服务调用必须覆盖超时、重试、取消和失败恢复。
- 性能结果应记录在测试报告、发布说明或约定位置。

## 18. 运行命令 `[通用 + 个性化]`

测试命令矩阵为：`{TEST_COMMANDS}`。

CI 必跑命令为：`{CI_TEST_COMMAND}`。

推荐命令分类：

```bash
# 运行全部默认测试
{CI_TEST_COMMAND}

# 运行单元测试
{UNIT_TEST_COMMAND}

# 运行集成测试
{INTEGRATION_TEST_COMMAND}

# 运行覆盖率
{COVERAGE_COMMAND}

# 运行前端检查
{FRONTEND_CHECK_COMMAND}

# 运行 E2E
{E2E_TEST_COMMAND}
```

生成项目文档时必须使用项目真实脚本；命令未知时写 `待确认`，不得编造。

## 19. CI 与质量门禁 `[条件启用]`

当项目存在 CI/CD、发布流程、PR 检查或自动化验收时启用本节。

- PR 合入前必须运行项目约定的默认测试命令。
- 正式功能无测试不应合入；如暂无法自动化测试，必须记录人工验证方式和原因。
- CI 应区分快速必跑测试、慢速测试、E2E、性能测试和兼容性测试。
- 测试报告、覆盖率报告、失败日志应可追溯。
- 发布前测试门禁应与 `rules/release.md` 保持一致。

## 20. AI 修改测试规则 `[通用]`

AI 修改代码时必须：

- 先判断变更属于新增功能、行为变更、Bug 修复、重构、文档更新还是配置变更。
- 行为变化时同步新增或更新测试；纯重构必须保证既有测试通过。
- 优先补充最小但有效的测试，覆盖真实风险路径。
- 保持测试文件路径与源文件路径镜像或符合项目约定。
- 运行与变更范围匹配的测试命令；无法运行时说明原因和残余风险。
- 不得删除、跳过、弱化测试来让任务通过。
- 不得使用真实外部服务、真实密钥、真实生产数据完成测试。

## 21. 与其他规则的同步关系 `[通用]`

| 变更类型 | 必须同步 |
| --- | --- |
| API、错误码、OpenAPI | `rules/api.md`、接口测试、契约测试、SDK 测试 |
| 认证、权限、安全 | `rules/security.md`、权限测试、安全回归测试 |
| 数据库、迁移、种子数据 | `rules/database.md`、Repository 测试、迁移测试 |
| 数据资产、fixtures、样例数据 | `rules/data-management.md`、测试数据说明 |
| 上传、媒体、对象存储 | `rules/media.md`、`rules/object-storage.md`、上传下载测试 |
| 环境变量、配置 | `rules/environment.md`、配置校验测试 |
| 兼容性、信创、多端 | `rules/compatibility.md`、兼容性测试矩阵 |
| 发布、回滚、部署验证 | `rules/release.md`、发布验收和回归测试 |

## 22. 初始化生成建议 `[通用]`

用于工程初始化生成 `testing.md` 时，建议按以下步骤处理：

1. 根据用户输入替换 `{PRODUCT_NAME}`、`{PRODUCT_CODE}`、测试栈、覆盖率、命令、数据库策略、Mock 策略等占位符。
2. 保留所有 `[通用]` 章节，并将模板语气改为项目确定性约束。
3. 对 `[个性化]` 章节填入项目真实测试框架、目录、命令和覆盖率目标；信息不足时标记 `待确认`。
4. 对 `[条件启用]` 章节按项目能力裁剪；未启用能力应删除或标记为“不适用”。
5. 与 `api.md`、`coding.md`、`database.md`、`data-management.md`、`security.md`、`media.md`、`object-storage.md`、`release.md` 保持一致。
6. 生成后检查是否仍包含其他项目业务词、虚构命令、虚构测试框架或无法执行的质量门禁。

## 23. 完成任务后检查清单 `[通用]`

- 是否新增或更新了与行为变更对应的测试。
- 是否覆盖成功路径、校验错误、业务错误、权限错误、外部服务失败和边界条件。
- 是否没有使用生产数据、真实密钥、真实外部服务或开发者本机状态。
- 是否测试名称、目录和 Fixture 符合项目约定。
- 是否运行了与变更范围匹配的测试命令。
- 是否无法运行的测试已说明原因和风险。
- 是否 API、数据库、环境、安全、发布相关测试与对应规则文档保持一致。
