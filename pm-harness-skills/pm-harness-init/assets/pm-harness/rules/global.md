# 全局规则

> 模块标记说明：
>
> - **[通用]**：适用于大多数 Harness 工程，初始化时默认保留。
> - **[个性化]**：必须根据用户项目输入生成，不能直接沿用模板默认值。
> - **[条件启用]**：只有项目具备对应能力时才保留或展开，例如 monorepo、前端、算法、对象存储、私有化部署。

## 0. 规则定位 [通用]

`rules/global.md` 是所有 AI Agent 的全局行为约束，用于回答四个问题：

- 当前任务是否允许直接执行。
- 执行前必须读取哪些上下文。
- 哪些行为绝对禁止。
- 完成前必须如何验证和汇报。

本文件只定义跨项目、跨模块、跨任务类型的全局规则。具体语言、代码、测试、安全、API、数据库、UI、部署、目录结构等细则，应继续读取对应 `rules/*.md`。

初始化生成本文件时，必须用用户输入替换占位符；缺失信息可以标记为 `待确认`，不得编造项目事实。

常用占位符：

| 占位符 | 含义 | 归属 |
|---|---|---|
| `{PRODUCT_NAME}` | 产品或项目名称 | [个性化] |
| `{PRODUCT_CODE}` | 项目代码或仓库 slug | [个性化] |
| `{PRODUCT_FORMS}` | 产品形态，如 Web、小程序、移动端、桌面端 | [个性化] |
| `{MODULES}` | 项目模块列表 | [个性化] |
| `{BACKEND_STACK}` | 后端技术栈 | [个性化] |
| `{FRONTEND_STACK}` | 前端技术栈 | [个性化] |
| `{DATABASE_STACK}` | 数据库与迁移方案 | [个性化] |
| `{DEPLOYMENT_STACK}` | 部署方式 | [个性化] |
| `{PRIMARY_VERIFY_COMMAND}` | 统一验证命令 | [个性化] |
| `{TASK_TRACKING_SYSTEM}` | 需求、BUG 或任务登记系统 | [个性化] |

## 1. 执行前提 [通用]

AI Agent 在执行任何任务前，必须完成以下检查：

1. 已读取 `AGENTS.md`。
2. 已读取本文件 `rules/global.md`。
3. 已确认当前任务类型：需求、BUG、代码修改、文档修改、测试、部署、数据、UI、算法、评审或运维。
4. 已确认当前任务影响范围：单文件、单模块、多模块、跨端、跨服务或全局规则。
5. 已读取当前任务相关的 `rules/*.md`。
6. 已检查工作区现有变更，避免覆盖用户未要求修改的文件。
7. 如为正式功能、BUG 修复或高风险变更，已确认存在或需要创建 OpenSpec Change。
8. 如涉及需求、BUG 或迭代，已确认对应 `issues/`、`iterations/` 或 `{TASK_TRACKING_SYSTEM}` 记录。

执行前不清楚任务边界时，应先通过仓库文件查证。只有在无法从本地上下文推断且继续执行会造成明显风险时，才向用户提问。

## 2. 任务分级 [通用]

| 级别 | 定义 | 示例 | 要求 |
|---|---|---|---|
| L0 轻量 | 不改变行为和契约的小改动 | 拼写、注释、README 小修、格式化 | 可直接编辑；仍需避免覆盖无关改动 |
| L1 文档 | 改长期文档但不改变工程规则 | 补充说明、更新操作步骤、整理索引 | 可直接编辑；若改变流程语义需升级 |
| L2 标准 | 单模块内功能、BUG、测试或配置变更 | 后端接口修复、前端页面调整、测试补充 | 需要相关 rules + 测试；正式能力需 OpenSpec Change |
| L3 跨模块 | 影响 2 个及以上模块或端 | API 契约、数据格式、SDK、前后端联动 | 必须说明影响面，同步契约、测试和文档 |
| L4 高风险 | 影响安全、数据、部署、状态机、权限、工作流 | 权限模型、迁移、部署拓扑、密钥、支付、license | 必须走 OpenSpec Change，并补充回滚和验证方案 |
| L5 治理变更 | 改 AGENTS、rules、目录边界、研发流程 | 新增顶层目录、修改 OpenSpec 流程 | 必须说明规则影响，更新相关文档和模板资产 |

任务级别只能上调，不能为了减少流程而下调。若任务同时满足多个级别，按最高级别处理。

## 3. 模块归属与上下文路由 [个性化 + 条件启用]

初始化时应根据项目结构生成模块路由。示例：

| 任务范围 | 路径 | 必读上下文 |
|---|---|---|
| 全局治理 | `AGENTS.md`、`rules/`、`docs/`、`openspec/` | `AGENTS.md`、`rules/global.md`、`rules/document-governance.md` |
| 后端 | `src/backend/` | `{BACKEND_CONTEXT}`、`rules/coding.md`、`rules/api.md`、`rules/database.md`、`rules/testing.md` |
| 前端 | `src/web/` 或 `frontend/` | `{FRONTEND_CONTEXT}`、`rules/ui-design.md`、`rules/api.md`、`rules/testing.md` |
| 小程序/移动端 | `{MOBILE_PATHS}` | `{MOBILE_CONTEXT}`、`rules/compatibility.md`、`rules/ui-design.md` |
| 算法/模型 | `src/algorithm/`、`models/` | `{ALGORITHM_CONTEXT}`、`rules/data-management.md`、相关契约文档 |
| 部署 | `deploy/`、`docker-compose.yml`、`.env.example` | `rules/environment.md`、`rules/port-management.md`、部署文档 |
| 数据与媒体 | `data/`、`models/`、对象存储配置 | `rules/data-management.md`、`rules/media.md`、`rules/object-storage.md` |

若项目不是 monorepo，可保留“全局治理 + 业务源码 + 部署 + 数据”四类；不要生成不存在的模块入口。

## 4. 禁止行为 [通用]

AI Agent 不得执行以下行为：

- 不得绕过 OpenSpec Change 直接开发正式功能。
- 不得直接修改 `openspec/specs/` 中的正式规范，归档合并动作除外。
- 不得在根目录随意新增业务代码。
- 不得绕过 `rules/directory-structure.md` 新增顶层目录或移动模块边界。
- 不得覆盖用户未要求修改的文件或无关改动。
- 不得读取、提交、输出 `.env`、token、secret、私钥、生产凭据、真实用户数据。
- 不得执行破坏性删除、强制 push、历史重写、仓库外写入，除非用户明确要求并批准。
- 不得引入隐藏状态源；需求、任务、规格、验证结果必须落到仓库约定文件或项目约定系统中。
- 不得在根治理层复制子项目业务代码。
- 不得为了本机临时问题随意修改应用内部端口、数据库类型、鉴权策略或部署拓扑。
- 不得编造不存在的命令、文件、接口、服务地址、测试结果或用户确认。

## 5. OpenSpec 与变更准入 [通用]

以下任务必须创建或更新 OpenSpec Change：

- 新增、修改或移除正式业务能力。
- 修复影响用户行为、数据一致性、安全、权限或接口契约的 BUG。
- 修改 API、事件、Webhook、SDK、文件格式、数据模型或跨模块契约。
- 修改数据库 schema、迁移、索引、种子数据规则或兼容性策略。
- 修改部署架构、环境变量、端口、容器、存储、队列或外部依赖。
- 修改 UI 设计系统、页面模板、核心交互或验收标准。
- 修改 AGENTS、rules、目录结构、工程流程或初始化模板。

以下任务通常可以不创建 OpenSpec Change：

- 不改变行为的拼写、格式、注释和链接修正。
- 单纯补充已有文档的说明文字。
- 不影响产品能力的测试用例补充。
- 明确由用户要求的临时探索或只读分析。

无法判断是否需要 OpenSpec Change 时，按“需要”处理，或在回复中明确风险和假设。

## 6. 状态源规则 [通用]

项目状态必须落在可审计位置，不能只存在于聊天记录。

| 状态类型 | 推荐状态源 | 说明 |
|---|---|---|
| 产品需求 | `issues/requirements/REQ-xxxx/` | PRD、用户故事、验收标准、trace |
| BUG | `issues/bugs/BUG-xxxx/` | 现象、根因、workaround、验收、证据 |
| 迭代 | `iterations/<sprint-id>/` | 范围、承诺、状态、验收、发布 |
| 工程变更 | `openspec/changes/<change-id>/` | proposal、design、specs、tasks |
| 已生效能力 | `openspec/specs/` | 归档后的正式规格 |
| 架构与契约 | `docs/architecture/`、`docs/specs/` | 长期有效的跨模块知识 |
| 验证结果 | `docs/reports/` 或 change/trace | 测试、构建、部署、验收证据 |
| 操作手册 | `docs/guides/` 或部署文档 | 可重复执行的步骤 |

如果项目使用外部任务系统，应在本文件中声明 `{TASK_TRACKING_SYSTEM}`，并在仓库内保留可追踪引用。

## 7. 文档与规则更新 [通用]

修改文档或规则时必须遵守：

- README 面向第一次打开仓库的人。
- AGENTS.md 面向 AI Agent，强调可执行、可验证、可路由。
- rules 文件面向强制约束，应短而明确，避免口号化。
- docs 文件面向长期知识，应说明背景、决策、影响和验证。
- issues 文件面向需求与 BUG 生命周期。
- iterations 文件面向交付计划和验收。
- openspec 文件面向能力变更和规格归档。

修改 AGENTS、rules 或初始化模板时，必须同步考虑：

- 当前模板目录 `pm-harness/`。
- 初始化 Skill 资产目录 `pm-harness-skills/pm-harness-init/assets/pm-harness/`。
- 如存在打包模板 zip，应同步重新打包或记录未更新原因。

## 8. 数据、安全与隐私底线 [通用]

AI Agent 必须遵守：

- 不读取 `.env`、私钥、token、生产配置、真实用户数据，除非用户明确要求且任务必要。
- 不把真实密钥、真实用户数据、内部凭据写入代码、文档、测试或示例。
- 示例配置只能使用占位符。
- 测试数据必须是假数据、脱敏数据或明确可提交的 fixtures。
- 运行时数据库、缓存、日志、上传文件和模型权重默认不提交。
- 涉及认证、授权、租户、角色、对象存储权限、文件上传时，必须读取 `rules/security.md`。

## 9. 验证准出 [通用 + 个性化]

完成任务前必须运行与变更范围匹配的验证命令。

推荐验证层级：

| 变更类型 | 必要验证 |
|---|---|
| 文档 | 链接、目录结构或文档检查（如项目提供） |
| 后端 | 单元测试、接口测试、类型/格式检查 |
| 前端 | lint、type-check、build、必要的 UI 验收 |
| API | OpenAPI 校验、客户端生成、契约测试 |
| 数据库 | migration 验证、回滚说明、兼容性测试 |
| 部署 | compose config、服务启动、健康检查 |
| UI | 视觉对照、截图、组件预览、可访问性基础检查 |
| 算法/模型 | 样例推理、输入输出契约、性能或资源说明 |
| 全局规则 | 检查相关模板、README、初始化资产是否同步 |

项目统一验证命令：

```bash
{PRIMARY_VERIFY_COMMAND}
```

如果验证无法运行，必须说明：

- 未运行的命令。
- 未运行原因。
- 已做的替代检查。
- 剩余风险。

不得声称未执行的验证已经通过。

## 10. 输出语言与回复要求 [通用 + 个性化]

AI 回复默认使用中文，除非用户明确要求其他语言。

代码命名默认使用英文。注释语言遵循 `rules/language.md` 和项目既有风格。

涉及修改时，回复必须包含：

- 修改了哪些文件。
- 为什么修改。
- 是否影响接口、数据库、UI、部署、安全或跨模块契约。
- 执行了哪些验证，结果如何。
- 哪些事项未验证或仍需人工确认。

涉及高风险或跨模块改动时，回复还必须包含：

- 影响范围。
- 回滚或恢复建议。
- 需要同步的文档、任务、规格或迭代记录。

## 11. 个性化强制规则 [个性化]

初始化时根据用户输入生成本节。示例：

```text
项目名称：{PRODUCT_NAME}
项目代码：{PRODUCT_CODE}
产品形态：{PRODUCT_FORMS}
模块列表：{MODULES}
后端技术栈：{BACKEND_STACK}
前端技术栈：{FRONTEND_STACK}
数据库：{DATABASE_STACK}
部署方式：{DEPLOYMENT_STACK}
任务系统：{TASK_TRACKING_SYSTEM}
统一验证命令：{PRIMARY_VERIFY_COMMAND}
```

如果用户未提供某项信息，应保留 `待确认`，并避免生成与该项强相关的硬性命令或目录。

## 12. 完成任务后全局检查清单 [通用]

```text
□ 是否已读取 AGENTS.md 和 rules/global.md
□ 是否已识别任务级别 L0-L5
□ 是否已读取当前任务相关 rules
□ 是否检查并保护用户已有无关改动
□ 是否需要 OpenSpec Change；如需要，是否已创建或更新
□ 是否更新需求、BUG、迭代或任务状态源
□ 是否遵守目录边界
□ 是否未读取或泄露密钥、真实用户数据、生产凭据
□ 是否同步更新文档、契约、测试或初始化模板
□ 是否运行相关验证命令
□ 是否在回复中说明验证结果和剩余风险
```

## 13. 初始化生成建议 [通用]

工程初始化工具生成 `rules/global.md` 时应：

1. 保留 [通用] 模块。
2. 用用户输入替换 [个性化] 占位符。
3. 根据项目能力保留或删除 [条件启用] 内容。
4. 从实际目录扫描生成模块路由，避免指向不存在的路径。
5. 从实际脚本生成验证命令，避免保留虚假命令。
6. 将未知项标记为 `待确认`。
7. 检查本文件是否能清楚回答：
   - 什么任务能直接做，什么任务必须走 OpenSpec？
   - 什么行为绝对禁止？
   - 做完以后如何证明完成？
