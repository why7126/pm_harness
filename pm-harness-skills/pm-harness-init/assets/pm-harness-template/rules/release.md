# 发布规范

## 0. 规则定位 [通用]

本文档用于约束 `{PRODUCT_NAME}` 项目的发布流程，覆盖：

- 发布准入条件。
- 版本号、Tag、镜像 Tag、构建产物管理。
- 后端、前端、移动端、算法、文档、数据库、部署包构建。
- 部署前后验证、冒烟测试、回归测试。
- 发布说明、变更归档、Issue/Iteration 状态同步。
- 回滚策略和应急处理。

AI 在参与发布、构建、部署、版本修改、Docker 镜像、CI/CD、回滚、发布说明生成时，必须读取本文档。

### 0.1 初始化占位符 [通用]

| 占位符 | 含义 | 生成要求 |
|---|---|---|
| `{PRODUCT_NAME}` | 产品中文名 | 来自用户输入 |
| `{PRODUCT_CODE}` | 项目代码名 | kebab-case |
| `{RELEASE_CHANNELS}` | 发布渠道 | dev、test、staging、production、private 等 |
| `{VERSION_STRATEGY}` | 版本策略 | SemVer、日期版本、Commit SHA 等 |
| `{BUILD_TARGETS}` | 构建目标 | backend、web、miniapp、mobile、algorithm、docs 等 |
| `{VERIFY_COMMANDS}` | 发布前验证命令 | 从实际脚本生成 |
| `{BUILD_COMMANDS}` | 构建命令 | 按技术栈生成 |
| `{DEPLOYMENT_STACK}` | 部署方式 | Docker Compose、Kubernetes、私有化、SaaS |
| `{DEPLOYMENT_VERIFY_CHECKLIST}` | 部署验证清单 | 按产品能力生成 |
| `{ROLLBACK_STRATEGY}` | 回滚策略 | 镜像、数据库、配置、前端、移动端等 |
| `{RELEASE_NOTE_TEMPLATE}` | 发布说明模板 | 按团队流程生成 |

生成初始化工程时，必须替换全部占位符；未知项写 `待确认`，不得保留样例项目的业务流程或命令。

## 1. 文档模块分类 [通用]

- `[通用]`：所有 Harness 工程默认保留。
- `[个性化]`：根据技术栈、部署模式、发布渠道、团队流程生成。
- `[条件启用]`：仅在对应能力启用时保留，例如 Web、移动端、算法、数据库迁移、对象存储、私有化、Kubernetes、应用商店。

未启用的发布目标不得出现在强制发布清单中。

## 2. 发布原则 [通用]

- 未通过质量门禁不得发布。
- 未同步文档、OpenSpec、Issue、Iteration 状态不得发布正式版本。
- 有数据库、配置、对象存储、接口契约变更时，必须具备回滚或补救方案。
- 生产发布必须使用可追溯版本号或确定性构建产物。
- 发布说明必须能回答：发布了什么、影响谁、如何验证、如何回滚。
- AI 不得凭空声明“已发布”“已构建成功”“已部署成功”，必须基于实际命令或用户提供结果。

## 3. 发布渠道 [通用 + 个性化]

发布渠道：

```text
{RELEASE_CHANNELS}
```

推荐矩阵：

| 渠道 | 用途 | 准入要求 | 产物 |
|---|---|---|---|
| `dev` | 开发联调 | 基础测试通过 | 本地服务/开发镜像 |
| `test` | 测试环境 | 单元/集成测试通过 | 测试镜像/部署包 |
| `staging` | 预发/验收 | 回归、兼容性、验收通过 | 候选发布产物 |
| `production` | 正式生产 | 发布审批、回滚方案、监控就绪 | 正式版本 |
| `private` | 私有化交付 | 客户环境清单、离线包、License | 私有化部署包 |

初始化时根据 `{DEPLOYMENT_STACK}` 和团队流程裁剪。

## 4. 发布准入检查 [通用 + 个性化]

发布前必须生成并执行：

```text
{VERIFY_COMMANDS}
```

推荐检查清单：

```text
□ 需求、Bug、OpenSpec Change 状态已确认
□ OpenSpec 校验通过
□ 单元测试通过
□ 集成测试通过
□ E2E/冒烟测试通过
□ API 契约和客户端生成已同步
□ 数据库迁移、回滚或补救方案已确认
□ .env.example 与部署环境变量已同步
□ 端口、对象存储、媒体、兼容性文档已同步
□ Docker 镜像或部署包构建成功
□ 发布说明已生成
□ 回滚策略已明确
□ 相关 Issue、Iteration、Release Note 已更新
```

具体命令必须来自项目实际脚本；未知时写 `待确认`，不得编造。

## 5. 版本策略 [通用 + 个性化]

版本策略：

```text
{VERSION_STRATEGY}
```

推荐规则：

| 类型 | 示例 | 适用场景 |
|---|---|---|
| SemVer | `1.2.3` | 产品化版本 |
| 日期版本 | `2026.06.16` | 内部系统/交付包 |
| RC 版本 | `1.2.3-rc.1` | 预发布 |
| Commit SHA | `sha-abcdef1` | 镜像、可追溯构建 |
| Build 编号 | `build.1234` | CI/CD |

要求：

- 后端、前端、移动端、算法、部署包的版本来源必须明确。
- Docker 镜像应使用不可变 Tag，例如语义化版本 + commit SHA。
- `latest` 只能作为辅助标签，不得作为生产唯一回滚依据。
- 版本变更必须同步发布说明和归档记录。

### 6.2 前端构建 [条件启用]

按 `{FRONTEND_STACK}` 生成：

```bash

```

要求：

- 构建时公开变量不得包含密钥。
- API Base URL、运行时配置、资源路径必须与部署模式一致。
- 构建产物应经过基础页面加载或静态资源检查。

### 6.3 移动端 / 小程序构建 [条件启用]

启用移动端、小程序、桌面端时，必须补充：

- 构建命令。
- 签名/证书管理方式。
- 应用商店或客户交付流程。
- 版本号和构建号规则。
- 回滚或撤回策略。

### 6.4 算法 / 模型构建 [条件启用]

启用算法、模型、推理服务时，必须补充：

- 镜像或部署包构建方式。
- 模型版本、校验和、下载路径。
- 推理接口兼容性验证。
- CPU/GPU/架构兼容性说明。

## 7. 镜像与制品管理 [通用 + 条件启用]

发布产物必须可追溯：

| 产物 | 命名建议 | 必须记录 |
|---|---|---|
| 后端镜像 | `{PRODUCT_CODE}-backend:{version}-{sha}` | 版本、commit、构建时间 |
| Web 镜像 | `{PRODUCT_CODE}-web:{version}-{sha}` | 版本、commit、构建参数 |
| 算法镜像 | `{PRODUCT_CODE}-algorithm:{version}-{sha}` | 模型版本、依赖 |
| 私有化包 | `{PRODUCT_CODE}-{version}-private.zip` | 校验和、客户/环境 |
| 文档包 | `{PRODUCT_CODE}-docs-{version}.zip` | 文档版本 |

禁止只依赖可变 Tag 进行生产部署。

## 8. 部署验证 [通用 + 个性化]

部署验证清单：

```text
{DEPLOYMENT_VERIFY_CHECKLIST}
```

推荐验证：

```text
□ 服务健康检查返回正常
□ 前端页面可加载
□ 登录/鉴权流程正常
□ 核心 API 可调用
□ 数据库连接正常
□ 对象存储上传/下载正常（如启用）
□ 文件/媒体处理流程正常（如启用）
□ 异步任务/队列正常（如启用）
□ 算法/模型推理正常（如启用）
□ 日志无启动级错误
□ 监控/告警可用（如启用）
```

健康检查路径、端口、服务名必须与 `api.md`、`port-management.md`、`environment.md` 一致。

## 9. 数据库发布规则 [条件启用]

涉及数据库变更时：

- 必须明确 migration 顺序。
- 必须评估兼容旧代码/新代码的窗口。
- 必须准备回滚 SQL、补偿脚本或人工修复方案。
- 必须备份重要数据或确认无需备份。
- 必须验证 seed 数据幂等。
- 多数据库或信创数据库必须验证兼容脚本。

不得在无迁移计划、无备份/补救方案时发布破坏性 Schema 变更。

## 10. 配置与环境发布规则 [通用]

新增或修改环境变量时：

- 必须同步 `.env.example`。
- 必须同步部署文档。
- 必须说明默认值、是否必填、是否敏感。
- 生产密钥必须通过密钥管理或部署平台注入。
- 不得在发布说明中泄露真实配置。

相关规则以 `rules/environment.md` 为准。

## 11. 发布说明 [通用 + 个性化]

发布说明模板：

```text
{RELEASE_NOTE_TEMPLATE}
```

推荐结构：

```markdown
# Release {version}

## 发布范围

## 新增能力

## 修复问题

## 兼容性影响

## 数据库/配置变更

## 部署步骤

## 验证结果

## 回滚方案

## 关联 Issue / OpenSpec / PR
```

发布说明必须引用相关 Issue、OpenSpec Change、Commit/PR、测试结果。

## 12. 回滚策略 [通用 + 个性化]

回滚策略：

```text
{ROLLBACK_STRATEGY}
```

推荐规则：

| 类型 | 回滚方式 |
|---|---|
| Docker 镜像 | 使用上一稳定镜像 Tag 回退 |
| 前端静态资源 | 回退上一构建产物或镜像 |
| 配置 | 回退上一版本环境配置 |
| 数据库 | 执行回滚 SQL、补偿脚本或恢复备份 |
| 对象存储 | 避免删除不可恢复对象，必要时恢复备份 |
| 移动端/小程序 | 按平台规则回滚/撤回/灰度 |
| 算法模型 | 切换上一模型版本 |

必须明确哪些变更可自动回滚，哪些需要人工介入。

## 13. 归档与状态同步 [通用]

发布完成后必须同步：

- OpenSpec Change 归档。
- `openspec/specs/` 更新。
- Issue 状态更新。
- Iteration release note / acceptance report 更新。
- README、docs、compatibility、deployment 文档更新。
- 发布说明保存到项目约定位置。

如果暂不能归档，必须说明阻塞原因和后续处理人。

## 14. 私有化 / 离线交付 [条件启用]

启用私有化或离线交付时，必须补充：

- 离线部署包组成。
- 镜像导出/导入方式。
- License 或证书处理。
- 客户环境依赖清单。
- 数据初始化步骤。
- 回滚包或上一版本保留策略。
- 验收清单。

客户密钥、生产数据、客户私有配置不得进入仓库。

## 15. CI/CD [条件启用]

启用 CI/CD 时，必须说明：

- 触发条件。
- 构建矩阵。
- 质量门禁。
- 制品发布位置。
- 部署目标。
- 人工审批节点。
- 密钥注入方式。
- 回滚方式。

CI/CD 配置变更必须同步本文档和部署文档。

## 16. AI 更新规则 [通用]

AI 修改以下内容时，必须检查发布规则同步：

| 修改内容 | 必须同步 |
|---|---|
| 构建命令 | release.md、README、CI/CD |
| Dockerfile / compose | release.md、deployment、environment、port-management |
| API 契约 | release.md、api.md、OpenAPI、前端客户端 |
| 数据库迁移 | release.md、database.md、回滚说明 |
| 环境变量 | release.md、environment.md、.env.example |
| 对象存储/媒体 | release.md、object-storage.md、media.md |
| 测试门禁 | release.md、testing.md、CI |
| 版本策略 | release.md、README、发布说明 |

AI 不得在未执行验证命令时声称验证通过。

## 17. 禁止事项 [通用]

- 禁止未通过测试和 OpenSpec 校验发布。
- 禁止发布说明缺失变更范围、验证结果或回滚方案。
- 禁止使用不可追溯产物发布生产版本。
- 禁止只改代码不更新版本、文档、发布说明。
- 禁止无回滚/补救方案发布破坏性数据库变更。
- 禁止在发布包中包含真实 `.env`、密钥、客户配置、测试缓存。
- 禁止保留样例项目业务发布流程、镜像名或命令。

## 18. 初始化生成建议 [通用]

将本文档作为工程初始化模块时，生成器应按以下步骤处理：

1. 根据 `{RELEASE_CHANNELS}` 生成发布渠道。
2. 根据 `{VERSION_STRATEGY}` 生成版本号、Tag、镜像 Tag 规则。
3. 根据 `{BUILD_TARGETS}` 和实际技术栈生成构建目标。
4. 根据实际脚本生成 `{VERIFY_COMMANDS}` 和 `{BUILD_COMMANDS}`；未知写 `待确认`。
5. 根据 `{DEPLOYMENT_STACK}` 生成 Docker Compose、Kubernetes、SaaS、私有化或离线交付规则。
6. 根据数据库、对象存储、媒体、算法、移动端能力保留对应 `[条件启用]` 章节。
7. 根据 `{ROLLBACK_STRATEGY}` 生成可执行回滚策略；未知写 `待确认`。
8. 检查本文档与 `testing.md`、`environment.md`、`database.md`、`api.md`、`compatibility.md`、`document-governance.md` 一致。

```text
□ 发布准入、验证命令、构建命令已参数化
□ [通用]、[个性化]、[条件启用] 模块标识完整
□ 发布渠道和版本策略符合项目实际
□ 构建目标只包含已启用服务/端
□ 部署验证清单覆盖核心能力
□ 数据库、环境变量、对象存储、媒体、算法等条件章节已按能力裁剪
□ 发布说明、回滚策略、归档同步规则完整
□ AI 更新规则覆盖构建、部署、测试、文档和版本
□ 未保留样例项目镜像名、命令或业务流程
```
